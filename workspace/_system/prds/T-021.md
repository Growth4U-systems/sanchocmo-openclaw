# PRD — T-021: Multi-client routing en crons

## Objetivo

Los cron jobs (daily pulse, heartbeats, reporting) deben iterar sobre todos los clientes activos, ejecutar con el contexto correcto de cada uno, y publicar resultados en el Discord correspondiente.

## Contexto y motivación

- Actualmente hay un solo cliente (Hospital Capilar) y los crons asumen contexto único.
- Al escalar a múltiples clientes, cada cron debe saber: qué clientes hay, dónde está su data, y a qué Discord canal publicar.
- `clients.js` ya existe como registro central de clientes.

## Requisitos funcionales

- **RF-1**: Registro central de clientes con metadata:
  ```json
  { "slug": "hospital-capilar", "name": "Hospital Capilar", "guild": "...", "channels": {...}, "active": true }
  ```
- **RF-2**: Crons leen `clients.js` (o `clients.json`) al inicio
- **RF-3**: Para cada cliente activo: cargar contexto (brand/, sources.json), ejecutar tarea, publicar en su canal
- **RF-4**: Si un cliente falla, no bloquea los demás (try/catch por cliente)
- **RF-5**: Log por cliente: qué se ejecutó, resultado, duración

## Archivos afectados

| Archivo | Acción |
|---------|--------|
| `clients.js` o `clients.json` | Modificar — schema estándar |
| Cron jobs existentes | Modificar — añadir loop por cliente |
| `scripts/new-client.sh` (T-017) | Modificar — registrar en clients.json |

## Pasos de implementación

1. Definir schema de clients.json (slug, name, guild, channels, active, paths)
2. Migrar clients.js actual a formato estándar
3. Crear helper function: `for_each_client(callback)` que itera y maneja errores
4. Refactorizar crons existentes para usar el helper
5. Test con Hospital Capilar como único cliente
6. Test con cliente dummy adicional

## Criterios de aceptación

- [ ] clients.json tiene schema estándar documentado
- [ ] Crons iteran sobre clientes activos
- [ ] Fallo en un cliente no bloquea los demás
- [ ] Resultados se publican en el canal correcto de cada cliente
- [ ] new-client.sh registra nuevo cliente automáticamente
- [ ] Log por cliente por ejecución

## Riesgos y dependencias

| Riesgo | Mitigación |
|--------|-----------|
| Dependencia de T-017 (new-client.sh) | Puede implementarse independientemente; T-017 lo mejora |
| Cron execution time crece linealmente con clientes | Paralelizar si >5 clientes; por ahora secuencial OK |
| Contexto incorrecto contamina otro cliente | Cada iteración carga contexto fresh, no reutiliza state |
