# PRD — T-010: Migrar Mission Control a Next.js

## Objetivo

Reemplazar el dashboard estático actual (`mission-control.html` + `regenerate.py` + `*.js`) por una app Next.js con lectura/escritura en vivo sobre los archivos del workspace, eliminando el ciclo de regeneración manual y habilitando interactividad real (CRUD de tareas, formularios, Foundation en tiempo real).

## Contexto y motivación

### Estado actual
- **Mission Control**: Un HTML monolítico de 2,259 líneas + 4 archivos JS de datos (`mc-data.js`, `clients.js`, `agents-data.js`, `skills-data.js`) + script Python de regeneración.
- **Regeneración**: Cada cambio en TASKS.md, brand/, memory/, etc. requiere ejecutar `python3 regenerate.py` para que el dashboard refleje la realidad. Es manual o vía heartbeat (cada ~2h). Los datos pueden estar desactualizados.
- **Interactividad**: Cero. No se puede crear/editar tareas, aprobar propuestas, ni escribir briefs desde el dashboard. Todo requiere volver a webchat o Discord.
- **Serving**: Tailscale serve sobre el archivo estático (`https://sancho-cmo.taild48df2.ts.net`).

### Gateway Dashboard vs Mission Control — ¿Fusionar o separar?

**Análisis:**

| Aspecto | Gateway Dashboard (Control UI) | Mission Control |
|---------|-------------------------------|-----------------|
| **Propósito** | Admin de OpenClaw: chat, nodos, config, exec approvals | Operaciones de marketing: tareas, Foundation, campañas, clientes |
| **Tech** | Vite + Lit, servido por el Gateway en `:18789/` | HTML estático + JS data files |
| **Auth** | WebSocket handshake con token/password | Tailscale serve con password |
| **Dueño** | OpenClaw core (actualizado con cada release) | SanchoCMO (nuestro, custom) |
| **Usuarios** | Admin técnico | CMO, marketing team |
| **Alcance** | Infraestructura del agente | Negocio del cliente |

**Recomendación: MANTENER SEPARADOS.** Razones:

1. **Upgrades sin miedo**: OpenClaw actualiza su Control UI con cada release. Si metemos lógica de marketing ahí, cada `npm update openclaw` puede romper nuestros cambios. Separar = cero conflictos.
2. **Dominios distintos**: El Gateway Dashboard es para "¿está el bot vivo?". Mission Control es para "¿cómo va la campaña de Hospital Capilar?". Mezclarlos diluye ambos.
3. **Multi-tenant**: Mission Control será multi-cliente. El Gateway Dashboard es per-instance. No escala igual.
4. **Autonomía de deploy**: Mission Control puede correr en Vercel, Cloudflare, o cualquier host. No depende del gateway running.

**Integración sin fusión:**
- Mission Control puede **leer del gateway** vía WebSocket/API para mostrar status (gateway running, discord connected, etc.) en un widget. Mejor de los dos mundos: datos de infra en el dashboard de marketing, sin acoplar código.
- Un link cruzado en ambas UIs: "Ir a Gateway Dashboard ↗" / "Ir a Mission Control ↗".

## Requisitos funcionales

### RF-01: API Routes — Lectura/escritura de archivos del workspace
- `GET /api/tasks` → parsea TASKS.md, devuelve JSON
- `PUT /api/tasks/:id` → edita una tarea en TASKS.md (status, notas, prioridad)
- `POST /api/tasks` → crea nueva tarea en TASKS.md
- `GET /api/foundation` → lee brand/ y calcula estado de pillars
- `GET /api/activity` → lee memory/*.md, devuelve feed
- `GET /api/campaigns` → lee campaigns/
- `GET /api/clients` → lee clients.js o directorio de clientes
- `GET /api/skills` → lee skills/ (nombre + descripción de cada SKILL.md)
- `GET /api/agents` → lee configuración de agentes
- `GET /api/system` → status del gateway, discord, tailscale (vía `openclaw status`)
- `GET /api/changelog` → parsea CHANGELOG.md
- `GET /api/prds/:id` → lee _system/prds/T-XXX.md
- `POST /api/prds/:id` → escribe/actualiza PRD

### RF-02: CRUD de tareas en vivo
- Kanban board con drag & drop entre columnas (Propuesta → Aprobada → En Progreso → Hecha)
- Click en tarea → modal con detalle completo + PRD si existe
- Botón "Aprobar" / "Rechazar" que escribe directamente en TASKS.md
- Formulario "Nueva tarea" que añade al board y al archivo
- Filtros por categoría ([infra], [skill], [agent]...) y prioridad (P0-P3)
- Auto-incremento de ID (lee último T-XXX, suma 1)

### RF-03: Skills reader con syntax highlighting
- Lista de skills con búsqueda
- Click → renderiza SKILL.md con syntax highlighting (código, tablas, headers)
- Diferenciar skills locales (SanchoCMO) vs OpenClaw built-in vs ClawHub
- Botón "Proponer cambio" que crea tarea tipo [skill]

### RF-04: Foundation progress en tiempo real
- DAG visual de los 14-16 pillars con dependencias (layers 0→5)
- Color por estado: ✅ done / 🔧 in progress / ⬜ pending / 🔒 blocked (dependencia no cumplida)
- Click en pillar → muestra contenido del archivo si existe, o "Pendiente — iniciar desde Discord"
- Progress bar global (X/14 completados)
- No requiere regenerar — lee brand/ en cada request

### RF-05: Formularios de escritura directa
- Crear tarea → escribe en TASKS.md
- Aprobar/rechazar tarea → muta TASKS.md
- Crear PRD → escribe en _system/prds/
- Editar notas de tarea → muta TASKS.md
- Añadir nota diaria → escribe en memory/YYYY-MM-DD.md

### RF-06: SSR para velocidad
- Server-side rendering de todas las páginas
- ISR (Incremental Static Regeneration) no aplica — los datos cambian con cada file write
- API routes con lectura directa de filesystem (no DB)
- Caché inteligente: invalidar cuando el archivo cambia (fs.watch o check mtime)

### RF-07: Tailscale serve mantenido
- El Next.js server escucha en un puerto local (ej: 3000)
- Tailscale serve proxy de `https://sancho-cmo.taild48df2.ts.net` → `localhost:3000`
- Auth: mantener password actual o migrar a Next.js middleware con auth propia
- PM2 o systemd para mantener el proceso vivo

### RF-08: Widget de status del Gateway
- Pequeño widget en el header/sidebar que muestra:
  - Gateway: 🟢 Running / 🔴 Stopped
  - Discord: 🟢 Connected / 🔴 Disconnected
  - Link: "Abrir Gateway Dashboard ↗"
- Datos via llamada a `openclaw status` o WebSocket al gateway

### RF-09: Multi-cliente
- Selector de cliente en sidebar (como el actual)
- Cada cliente tiene su directorio de brand/, campaigns/, etc.
- Las vistas de Foundation, Campaigns, Activity se filtran por cliente seleccionado
- Vista global: resumen de todos los clientes

### RF-10: Dark/Light mode
- Mantener el theming actual (CSS variables)
- Persistir preferencia en localStorage
- Respetar `prefers-color-scheme` del sistema

## Riesgos y dependencias

| Riesgo | Impacto | Mitigación |
|--------|---------|-----------|
| **Escritura concurrente en TASKS.md** | Si Sancho edita TASKS.md desde chat al mismo tiempo que la UI, se pueden pisar cambios | Escritura atómica con lock file (`tasks.lock`). Leer-modificar-escribir con retry. |
| **Node.js en el Mac 24/7** | Next.js server debe estar siempre corriendo | PM2 con auto-restart + LaunchAgent como el gateway |
| **Parsing de TASKS.md frágil** | El formato Markdown de las tablas es estricto; un typo rompe el parser | Tests unitarios para el parser. Validación antes de escribir. Backup antes de mutar. |
| **Tailscale serve conflict** | Ya hay un serve activo para el HTML; cambiar a Next.js puede requerir reconfigurarlo | Script de migración que actualiza el serve target |
| **Dependencia de filesystem** | Next.js en Vercel no puede leer filesystem local | No deployer en Vercel. Self-hosted only (localhost + Tailscale). Si algún día se quiere cloud, migrar a Supabase como backend. |
| **Compatibilidad con futuras versiones de OpenClaw** | Si OpenClaw cambia la estructura del workspace | Workspace paths centralizados en `lib/workspace.ts`. Fácil de actualizar. |
| **Scope creep** | Tentación de añadir features durante el build | Respetar las 6 fases. Features extra → nuevas tareas en TASKS.md. |
