# PRD — T-023: Sistema de selección de modelos multi-tier

## Objetivo

Asignar modelos por tier a cada agente y cron según la complejidad de la tarea, reduciendo costes un 50-60% sin perder calidad en las decisiones estratégicas.

## Contexto y motivación

- Actualmente TODOS los agentes usan Opus 4.6 ($15/$75 por MTok) — incluso Escudero que hace tareas mecánicas.
- 4 tiers disponibles con diferencia de precio 100x entre el más caro y el más barato:
  - **Tier 1 (Opus)**: $15/$75 — razonamiento complejo, estrategia
  - **Tier 2 (Sonnet)**: $3/$15 — ejecución con contexto, escritura
  - **Tier 3 (Haiku)**: $0.80/$4 — tareas simples, clasificación
  - **Tier 4 (MiniMax)**: $0.30/$1.10 — traducción, formato, extracción
- Asignación inteligente: cada agente/tarea usa el tier mínimo que garantiza calidad.

## Requisitos funcionales

- **RF-1**: Configuración de modelo por agente en openclaw.json:
  ```
  cervantes → Opus (arquitectura, código)
  sancho → Opus (estrategia CMO, decisiones)
  rocinante → Sonnet (QA, review — no necesita razonamiento profundo)
  escudero → Sonnet (ejecución de tareas)
  ```
- **RF-2**: Configuración de modelo por cron:
  - Heartbeats → Haiku (checks simples)
  - Daily pulse → Sonnet (extracción + clasificación)
  - Reports → Sonnet (compilación)
  - Foundation → Opus (estrategia)
- **RF-3**: Fallback chain: si tier falla → sube al siguiente
  - Haiku falla → Sonnet → Opus
- **RF-4**: Override por sesión: `/model opus` para forzar tier alto cuando necesario
- **RF-5**: Documentar qué tier usar para qué tipo de tarea (guía en BRAIN.md)

## Pasos de implementación

1. Definir matriz agente × tier (qué modelo para qué agente)
2. Actualizar openclaw.json: `model.primary` por agente
3. Actualizar cron jobs con modelo específico
4. Configurar fallback chain (si OpenClaw lo soporta nativamente, o via retry logic)
5. Documentar en BRAIN.md: cuándo usar cada tier
6. Test: verificar que cada agente usa su modelo asignado
7. Medir ahorro real vs baseline (T-022 proporciona los datos)

## Criterios de aceptación

- [ ] Cada agente usa su modelo asignado (verificar en `openclaw status`)
- [ ] Crons usan modelos apropiados al tipo de tarea
- [ ] Fallback funciona: si tier bajo falla, escala automáticamente
- [ ] `/model opus` override funciona por sesión
- [ ] BRAIN.md documenta guía de selección
- [ ] Ahorro estimado medible via T-022

## Riesgos y dependencias

| Riesgo | Mitigación |
|--------|-----------|
| Sonnet no tiene calidad suficiente para QA (Rocinante) | Test A/B: comparar output Opus vs Sonnet en 10 reviews; ajustar si necesario |
| Haiku pierde contexto en heartbeats complejos | Heartbeats deben ser simples por diseño; si necesitan contexto → Sonnet |
| MiniMax no soporta tools | Verificar compatibilidad antes de asignar; puede que Haiku sea el floor |
| Depende de T-022 para medir impacto | Puede implementarse sin T-022; el ahorro teórico es calculable |
