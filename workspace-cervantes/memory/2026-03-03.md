## T-044: Debug SIGTERM niche-discovery-100x (15:05)

### Admin Request de Alfonso via #soporte

**Problema reportado**: Scripts Python (serp_search.py, scrape_urls.py, extract_problems.py) se cuelgan y son matados con SIGTERM. No producen output.

### Investigación realizada

1. **Red/API: OK** — `curl` a google.serper.dev: 0.25s. Python urllib: 0.82s con respuesta exitosa.
2. **API Keys: OK** — SERPER_API_KEY (40 chars), FIRECRAWL_API_KEY (35 chars), OPENROUTER_API_KEY (73 chars) en .env.
3. **Exec timeout: NO es la causa** — Default 1800s (30 min). No overrides en config de agentes.
4. **Scripts: FUNCIONAN** — serp_search.py completó 125 queries (config reducido) en ~2 min. 31 URLs únicas. 0 errores.

### Causa raíz: Gateway restarts

**Los scripts mueren porque el Gateway recibe SIGTERM frecuentemente y mata todos los procesos hijo.**

Evidencia en logs:
- Feb 25: 1 restart
- Feb 26: 10 restarts
- Feb 27: 6 restarts
- Feb 28: 9 restarts
- Mar 01: 6 restarts
- Mar 02: 6 restarts
- Mar 03: 2 restarts (hasta las 15:00)

Causas de los restarts:
- Config patches (auto-binding Discord users)
- Auto-updates (openclaw control-ui triggered update.run)
- SIGUSR1 restarts (service restart)
- Discord health-monitor: **49 restarts solo el 3 de marzo** ("stuck"/"stale-socket" cada 10 min)

### Flujo del bug:
1. Sancho invoca `exec(command="python3 serp_search.py ...")`
2. Script arranca como child process del Gateway
3. Gateway recibe SIGTERM (por cualquiera de las causas arriba)
4. Gateway shutdown → `killProcessTree()` → SIGTERM a todos los hijos → SIGKILL tras grace period
5. Script muere sin producir output (o con output perdido si fue mid-turn)

### Por qué "sin output":
- Si el gateway restart ocurre en los primeros segundos, el script no llega ni al primer print
- O si el script ya producía output pero la sesión exec se pierde con el restart

### Recomendaciones
1. **Para scripts largos**: Ejecutar con `nohup` o en `tmux`/`screen` en vez de exec tool directo
2. **O usar**: `exec(command="...", background=true)` + `process(poll)` — el exec tool backgroundea el proceso, pero si el gateway reinicia, igual muere
3. **Fix real**: Investigar por qué Discord health-monitor reinicia 49 veces/día ("stuck") — T-045 potencial
4. **Workaround actual es válido**: usar harvest de Foundation + web_search del agente
