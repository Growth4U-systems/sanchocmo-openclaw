# PRD — T-040: Intelligence Log (histórico completo + visualización en MC)

## Objetivo

Evolucionar el tracker simple de deduplicación (`intelligence-tracker.json`) a un **Intelligence Log** completo que:
1. Almacena metadatos ricos de cada meeting y daily pulse procesado
2. Es visualizable en Mission Control como tabla con filtros y búsqueda
3. Mantiene la deduplicación (no reprocesar lo ya reportado)

## Decisión de almacenamiento: JSON rico (no Supabase)

**Por qué JSON y no Supabase:**
- Supabase no se usa activamente aún (T-018 descartada)
- MC actual es estático + mc-data.js — no hay backend SQL
- El volumen es bajo (~1 daily pulse/día + ~2-3 meetings/semana)
- regenerate.py ya sabe leer JSONs del workspace
- Cuando T-010 (Next.js) se implemente, se puede migrar trivialmente a Supabase

**Archivo:** `_system/intelligence-log.json`

### Campos

| Campo | Tipo | Requerido | Descripción |
|-------|------|-----------|-------------|
| `id` | string | ✅ | Unique key para deduplicación: `{type}-{date}-{slug}` |
| `type` | enum | ✅ | `meeting` \| `daily-pulse` |
| `client` | string | ✅ | Slug del cliente |
| `date` | string | ✅ | YYYY-MM-DD |
| `title` | string | ✅ | Título legible |
| `summary` | string | ✅ | Resumen breve (1-3 frases) |
| `status` | enum | ✅ | `processed` \| `skipped` \| `error` |
| `discordMessageId` | string | ❌ | ID del mensaje en Discord |
| `discordChannel` | string | ❌ | Canal donde se publicó |
| `discordMessageUrl` | string | ❌ | Link directo al mensaje |
| `sourceFile` | string | ❌ | Path relativo al archivo fuente |
| `processedAt` | string | ✅ | ISO 8601 timestamp |
| `insightsCount` | number | ❌ | Solo para daily-pulse |
| `tags` | string[] | ❌ | Tags libres para filtrado |

## Cambios en Mission Control

### Nueva sección: "📡 Intelligence Log"

Visible en vista admin (global) y vista cliente (filtrado por slug).

**Ubicación:** Después de la sección de Foundation, antes de Skills.

**Componentes:**
1. **Barra de búsqueda** — filtra por título o summary (client-side JS)
2. **Filtros** — por tipo (meeting/pulse), por cliente, por fecha (rango), por status
3. **Tabla/lista** — cada fila muestra:
   - 📅 Fecha
   - 🏷️ Tipo (badge: meeting=azul, pulse=verde)
   - 📝 Título
   - 💬 Resumen (truncado, expandible)
   - 🔗 Link a Discord (si existe)
   - 📄 Link a archivo fuente (si existe)
   - ✅ Status badge
4. **Ordenación** — por fecha (desc por defecto)
5. **Contador** — "23 entries · 15 meetings · 8 pulses"

### Estilo: Comic UI

- Tabla con bordes ink (3px), fondo parchment
- Badges tipo ComicStamp para tipo y status
- Búsqueda con borde ink
- Filtros como ComicButton toggle

## Instrucciones para Sancho (skills/crons)

Cuando Meeting Intelligence o Daily Pulse procesen algo:

1. Leer `_system/intelligence-log.json`
2. Comprobar si `id` existe en `entries[]`
3. Si existe → SKIP
4. Si no existe → procesar → añadir entry con todos los campos al array
5. Escribir el JSON actualizado

**Importante:** El `summary` lo genera Sancho al procesar. El `discordMessageUrl` se captura del output de `message(action=send)`.
