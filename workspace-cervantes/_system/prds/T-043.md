# PRD — T-043: Tablas estratégicas en Supabase

## Objetivo

Crear 6 tablas nuevas en Supabase para almacenar datos estratégicos relacionales (competidores, nichos, value criteria, assets, messaging) que hoy viven en markdown pero tienen relaciones many-to-many naturales imposibles de consultar en archivos planos.

**Por qué ahora:** El `positioning-messaging/references/schema.md` ya define estos datos como "Tier 2 databases" con IDs, foreign keys y relaciones. El schema relacional ya existe conceptualmente — falta materializarlo en Supabase.

## Arquitectura: Híbrido con Ownership Claro

Cada dato tiene **UN owner**. No hay duplicación ni sincronización.

| Tipo de dato | Owner | Ejemplo |
|---|---|---|
| Campos discretos (score, nombre, URL, enum, ID) | **Supabase** | `competitors.avg_review_score = 4.8` |
| Texto narrativo libre (análisis, research, copy largo) | **Markdown** | Battle cards en `competitors/current.md` |

### Supabase es owner de:
- Perfiles de competidores (campos estructurados, scores, tiers)
- Nichos/ECPs (scores, estado, metadata)
- Value Criteria con scores competitivos (shared cross-niche)
- Assets con conexiones a nichos y criteria
- Messaging entries (UVP/USPs con links a criteria)
- Relaciones many-to-many entre todas las entidades

### Markdown es owner de:
- Battle cards completas de competidores (narrativa profunda, 3 lenses)
- Investigación de niche-discovery (inventario de problemas, análisis JTBD, raw data)
- Positioning playbooks narrativos (el wrapper contextual)
- Brand voice, visual identity, company context
- Research crudo, deep dives, análisis extensos

### Puente entre ambos
- Markdown referencia IDs de Supabase: `<!-- supabase:competitor_id=3 -->`
- Skills escriben datos estructurados a Supabase y narrativa a markdown en el mismo flujo
- Queries de datos van a Supabase (via MCP). Contexto narrativo se lee de markdown

## Nuevas tablas (6 tablas + 1 junction)

### 1. `competitors` (master de competidores)

Complementa la tabla existente `competitor_moves` (que trackea señales/movimientos en tiempo real).

```sql
CREATE TABLE competitors (
  id SERIAL PRIMARY KEY,
  client_slug TEXT NOT NULL,
  name TEXT NOT NULL,
  url TEXT,
  competitor_type TEXT CHECK (competitor_type IN ('type_A', 'type_B', 'type_C', 'type_D')),
  monitoring_tier TEXT CHECK (monitoring_tier IN ('A', 'B', 'C')),
  positioning TEXT,
  key_claims JSONB DEFAULT '[]',
  avg_review_score NUMERIC(2,1),
  review_count INTEGER,
  strengths JSONB DEFAULT '[]',
  weaknesses JSONB DEFAULT '[]',
  synthesis TEXT,
  pricing_model TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(client_slug, name)
);

CREATE INDEX idx_competitors_client ON competitors(client_slug);
CREATE INDEX idx_competitors_tier ON competitors(monitoring_tier);
```

**Relación con tabla existente:** `competitor_moves.competitor_name` se une con `competitors.name` para conectar señales con perfiles.

### 2. `niches` (ECPs / segmentos)

```sql
CREATE TABLE niches (
  id SERIAL PRIMARY KEY,
  client_slug TEXT NOT NULL,
  name TEXT NOT NULL,
  persona TEXT,
  problem_core TEXT,
  pain_score INTEGER CHECK (pain_score BETWEEN 1 AND 10),
  reach_score INTEGER CHECK (reach_score BETWEEN 1 AND 10),
  size_score INTEGER CHECK (size_score BETWEEN 1 AND 10),
  total_score NUMERIC GENERATED ALWAYS AS (pain_score + reach_score + size_score) STORED,
  status TEXT DEFAULT 'active',
  created_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(client_slug, name)
);

CREATE INDEX idx_niches_client ON niches(client_slug);
CREATE INDEX idx_niches_status ON niches(status);
```

### 3. `value_criteria` (Tier 2, shared cross-niche)

```sql
CREATE TABLE value_criteria (
  id SERIAL PRIMARY KEY,
  client_slug TEXT NOT NULL,
  criterion TEXT NOT NULL,
  dimension TEXT CHECK (dimension IN (
    'compliance_reliability', 'operational_efficiency',
    'financial_intelligence', 'ecosystem_connectivity', 'emotional_drivers'
  )),
  relevance TEXT,
  justification TEXT,
  opportunity_type TEXT CHECK (opportunity_type IN ('red_ocean', 'no_market', 'opportunity_zone')),
  analysis TEXT,
  scores JSONB DEFAULT '{}',
  created_for_niche_id INTEGER REFERENCES niches(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_vc_client ON value_criteria(client_slug);
CREATE INDEX idx_vc_opportunity ON value_criteria(opportunity_type);
CREATE INDEX idx_vc_dimension ON value_criteria(dimension);
```

**Nota sobre `scores`:** JSONB con formato `{"Svenson": 4, "Insparya": 3, "client": 5, "diy": 2, "do_nothing": 1}`. El número variable de competidores hace que JSONB sea más práctico que normalizar aquí.

### 4. `value_criteria_niches` (junction many-to-many)

```sql
CREATE TABLE value_criteria_niches (
  criterion_id INTEGER REFERENCES value_criteria(id) ON DELETE CASCADE,
  niche_id INTEGER REFERENCES niches(id) ON DELETE CASCADE,
  PRIMARY KEY (criterion_id, niche_id)
);
```

### 5. `assets` (Tier 2, shared cross-niche)

```sql
CREATE TABLE assets (
  id SERIAL PRIMARY KEY,
  client_slug TEXT NOT NULL,
  asset TEXT NOT NULL,
  category TEXT CHECK (category IN ('qualifier', 'differentiator')),
  justification TEXT,
  created_for_niche_id INTEGER REFERENCES niches(id),
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_assets_client ON assets(client_slug);
CREATE INDEX idx_assets_category ON assets(category);
```

### 6. `asset_niche_connections` (many-to-many con datos extra)

```sql
CREATE TABLE asset_niche_connections (
  id SERIAL PRIMARY KEY,
  asset_id INTEGER REFERENCES assets(id) ON DELETE CASCADE,
  niche_id INTEGER REFERENCES niches(id) ON DELETE CASCADE,
  value_criterion_id INTEGER REFERENCES value_criteria(id),
  competitive_advantage TEXT,
  user_benefit TEXT,
  proof TEXT,
  proof_type TEXT CHECK (proof_type IN (
    'testimonial', 'screenshot', 'tutorial',
    'ad_message', 'comparative_table', 'report', 'case_study'
  )),
  UNIQUE(asset_id, niche_id, value_criterion_id)
);

CREATE INDEX idx_anc_asset ON asset_niche_connections(asset_id);
CREATE INDEX idx_anc_niche ON asset_niche_connections(niche_id);
```

### 7. `messaging` (UVP + USPs por nicho)

```sql
CREATE TABLE messaging (
  id SERIAL PRIMARY KEY,
  client_slug TEXT NOT NULL,
  niche_id INTEGER REFERENCES niches(id) ON DELETE CASCADE,
  message_category TEXT NOT NULL,
  hypothesis TEXT,
  value_criterion_id INTEGER REFERENCES value_criteria(id),
  objective TEXT,
  message_en TEXT,
  message_es TEXT,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX idx_messaging_client ON messaging(client_slug);
CREATE INDEX idx_messaging_niche ON messaging(niche_id);
CREATE INDEX idx_messaging_category ON messaging(message_category);
```

```
  └──> competitor_moves (EXISTING - real-time signals)
       (join on competitor_name = name)

niches (NEW)
  ├──> value_criteria_niches ──> value_criteria (NEW)
  │                                  └──> scores (JSONB, refs competitors by name)
  ├──> asset_niche_connections (NEW) ──> assets (NEW)
  │         └──> value_criterion_id ──> value_criteria
  └──> messaging (NEW)
           └──> value_criterion_id ──> value_criteria
```

## Queries que se desbloquean

```sql
-- ¿Qué assets son diferenciadores en 3+ nichos?
SELECT a.asset, COUNT(DISTINCT anc.niche_id) as niche_count
FROM assets a
JOIN asset_niche_connections anc ON a.id = anc.asset_id
WHERE a.category = 'differentiator' AND a.client_slug = 'hospital-capilar'
GROUP BY a.asset
HAVING COUNT(DISTINCT anc.niche_id) >= 3;

-- ¿Qué criteria son Opportunity Zone en todos los nichos?
SELECT criterion, dimension
FROM value_criteria
WHERE opportunity_type = 'opportunity_zone'
  AND client_slug = 'hospital-capilar';

-- Dame todos los USPs en español para un nicho
SELECT m.message_category, m.message_es, vc.criterion
FROM messaging m
LEFT JOIN value_criteria vc ON m.value_criterion_id = vc.id
JOIN niches n ON m.niche_id = n.id
WHERE n.name = 'El Frustrado del Minoxidil'
  AND m.message_category != 'UVP Core Promise';

-- ¿Qué competidores tienen score > 4 en criterio X?
SELECT c.name, (vc.scores->>c.name)::int as score
FROM competitors c, value_criteria vc
WHERE vc.criterion = 'Diagnóstico integral'
  AND (vc.scores->>c.name)::int > 4
  AND c.client_slug = vc.client_slug;

-- ¿Qué messaging angles no se han usado en campañas?
SELECT m.message_category, m.message_es, n.name
FROM messaging m
JOIN niches n ON m.niche_id = n.id
LEFT JOIN campaigns camp ON camp.name ILIKE '%' || m.message_category || '%'
WHERE camp.id IS NULL
  AND m.client_slug = 'hospital-capilar';
```
