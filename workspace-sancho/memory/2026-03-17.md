## Mission Control — Portal & Access Control (Mar 11-17)

### Token-based Access (implemented Mar 11)
- **Portal mode**: `/mc/portal/{mcToken}/` — client sees only their data
- **Admin mode**: `/mc/admin/{adminToken}/` — full access
- **Landing**: `/mc/` → generic page "Usa tu enlace de acceso"
- All direct routes (docs, APIs, connect pages) return 403 without token
- Tokens in `clients.json`: `mcToken` per client, `adminToken` at root
- Protocol documented in `_system/mc-links-protocol.md`

### Portal Features
- Serves real `mission-control.html` (not a separate dashboard)
- `clients.js` filtered to single client only
- `mc-data.js` parsed & filtered: only that client's foundation data, no costs
- Admin-only nav items hidden (Agents, Skills, Activity, Changelog)
- Client selector auto-set and hidden
- Dangerous APIs blocked: `/api/env`, `/api/restart-gateway`
- Allowed: `/api/client-integrations`, `/api/health-check`, `/api/metrics`, `/api/system-sa`
- Connect pages work with URL rewriting to portal base

### SOUL.md & TOOLS.md Updated
- Rule 3 "Links, nunca rutas": now requires token-based URLs
- Connect API section: tokenized URLs (portal for client guild, admin for internal guild)
- Sancho auto-detects guild → maps to correct token

- Applied to all rewrite points (admin wrapper, portal wrapper, JS file serving, connect pages)
