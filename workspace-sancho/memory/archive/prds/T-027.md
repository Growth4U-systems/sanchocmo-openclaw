# PRD — T-027: Docsify + Cloudflare Pages para docs fundacionales

## Objetivo

Montar un sitio de documentación con Docsify sobre `brand/` y hostearlo en Cloudflare Pages (gratis), protegido con Cloudflare Access. El agente al terminar un documento dice "Consúltalo aquí: [link]".

## Contexto y motivación

- Los documentos fundacionales (company-context, positioning, competitors, voice-profile, etc.) viven en archivos markdown en `brand/`.
- Actualmente solo los ve quien tiene acceso al workspace (Alfonso via Tailscale/webchat).
- Para compartir con el equipo del cliente se necesita una URL pública pero protegida.
- Tailscale requiere invitar gente y pagar por seats extra.
- Cloudflare Pages es gratis, Cloudflare Access permite proteger con email/OTP sin coste para <50 usuarios.
- Docsify convierte markdown a sitio navegable sin build step — los .md se sirven directamente.

## Requisitos funcionales

- **RF-1**: Docsify configurado sobre `brand/` como source
  - `index.html` con config de Docsify
  - `_sidebar.md` auto-generado desde la lista de archivos en brand/
  - Theme coherente con SanchoCMO (colores, tipografía)
- **RF-2**: Deploy automático a Cloudflare Pages
  - Opción A: Git push → Cloudflare Pages auto-deploy
  - Opción B: Wrangler CLI direct upload desde cron/script
- **RF-3**: Cloudflare Access protege el sitio
  - Política: solo emails autorizados (dominio del cliente + growth4u.io)
  - Login via email OTP (sin password)
- **RF-4**: Al completar un documento, Sancho incluye link: "📄 Consúltalo: https://docs-hospitalcapilar.pages.dev/company-context"
- **RF-5**: Sidebar refleja documentos disponibles (auto-actualizada)
- **RF-6**: Multi-cliente: cada cliente tiene su propio site (subdominio o path)
  - `docs-hospitalcapilar.pages.dev`
  - `docs-clienteb.pages.dev`

## Pasos de implementación

1. **Setup Docsify local**
   - Crear `brand/index.html` con CDN de Docsify
   - Configurar tema, search plugin, sidebar
   - Probar local: `npx docsify-cli serve brand/`

2. **Generar sidebar automática**
   - Script Python que lee `brand/*.md`, excluye `_sidebar.md` y `README.md`
   - Genera `_sidebar.md` con links a cada documento
   - Ejecutar como parte de regenerate.py o cron

3. **Setup Cloudflare Pages**
   - Crear proyecto en Cloudflare Pages (dashboard o CLI)
   - Configurar source: direct upload (no git — los docs no están en un repo)
   - Deploy inicial con wrangler: `wrangler pages deploy brand/ --project-name docs-hospitalcapilar`

4. **Configurar Cloudflare Access**
   - Crear application en Zero Trust dashboard
   - Política: Allow emails ending in `@hospitalcapilar.com` + `@growth4u.io`
   - Access method: One-time PIN (email OTP)

5. **Integrar en flujo de Sancho**
   - Cuando Sancho completa un documento de Foundation → auto-deploy + enviar link
   - Script: generate-sidebar → deploy-docs.sh → mensaje con URL

6. **Test end-to-end**
   - Crear doc → deploy → verificar que es visible → verificar que Access bloquea sin auth

## Riesgos y dependencias

| Riesgo | Mitigación |
|--------|-----------|
| Cloudflare Access requiere cuenta Cloudflare | Crear con alfonso@growth4u.io; free tier cubre todo |
| Wrangler CLI necesita auth token | Configurar `CLOUDFLARE_API_TOKEN` en .env |
| Docs contienen info sensible del cliente | Cloudflare Access + email OTP protege; no es público |
| Deploy frecuente durante Foundation (muchos docs) | Debounce: deploy máximo 1x por hora, o batch al final de sesión |
| Docsify no tiene build step — SEO pobre | No importa, es documentación interna, no necesita SEO |
