# PRD — T-028: Foundation iterativa con aprobación por pilar

## Objetivo

Cambiar la Foundation de "entrega de golpe" a flujo iterativo: un documento a la vez, en orden de dependencias, con presentación → validación → corrección → aprobación antes de pasar al siguiente.

## Contexto y motivación

- Actualmente foundation-orchestrator ejecuta varios pilares y entrega 7+ archivos de una vez con "✅ completado".
- El usuario no revisa, no corrige, no valida. Los datos quedan con errores o incompletos.
- Un CMO real presenta cada pieza de análisis, espera feedback, ajusta, y solo entonces avanza.
- La Foundation es la base de TODO lo que viene después (channels, content, GTM). Si está mal, todo está mal.

## Requisitos funcionales

- **RF-1**: Flujo ordenado por dependencias (Layer 0 → Layer 1 → Layer 2):
  - L0: company-context, budget-constraints (en paralelo)
  - L1: business-model-audit, self-intelligence (dependen de L0)
  - L2: market-intelligence, competitor-intelligence, niche-discovery, positioning-messaging, etc.
- **RF-2**: Para cada pilar:
  1. Sancho presenta resumen del documento (no el doc entero — un executive summary)
  2. Pregunta: "¿Esto está bien? ¿Quieres corregir algo?"
  3. Si corrección → ajusta → presenta de nuevo
  4. Si OK → marca como aprobado → guarda → siguiente
- **RF-3**: Estado persistente en `memory/foundation-state.json`:
  ```json
  { "company-context": "approved", "budget-constraints": "pending-review", "business-model-audit": "not-started" }
  ```
- **RF-4**: Retomable: si la sesión se corta, el siguiente mensaje retoma donde quedó
- **RF-5**: Progreso visible: "Foundation: 5/14 pilares completados ████░░░░░░ 36%"
- **RF-6**: Skip opcional: "Este pilar no aplica" → marcar como skipped con razón
- **RF-7**: Al completar todos: resumen ejecutivo de la Foundation completa

## Pasos de implementación

1. Definir orden de pilares con dependencias (DAG)
2. Diseñar formato de presentación/resumen por pilar
3. Reescribir foundation-orchestrator SKILL.md con flujo iterativo
4. Implementar state tracking en foundation-state.json
5. Implementar lógica de retomada (leer state → ir al siguiente pendiente)
6. Implementar progress bar y resumen final
7. Test con Hospital Capilar (que ya tiene datos parciales)

## Criterios de aceptación

- [ ] Sancho presenta UN pilar a la vez con resumen ejecutivo
- [ ] Usuario puede aprobar, corregir, o skipear cada pilar
- [ ] Correcciones se aplican antes de guardar
- [ ] Progreso visible en cada paso
- [ ] Sesión retomable tras interrupción
- [ ] Foundation completa genera resumen ejecutivo
- [ ] Dependencias respetadas (no presenta L2 sin L0 aprobado)

## Learnings integrados (2026-02-26)

### Persistence Protocol — foundation-orchestrator no menciona Supabase
Toda la referencia actual es a "Context Lake" y archivos `./brand/`. No hay instrucciones de cuándo/cómo escribir a Supabase. Añadir sección "Persistence Protocol" al SKILL.md:

| Dato | Destino | Por qué |
|------|---------|---------|
| Documentos de contexto (company-context, positioning, voice...) | `./brand/*.md` | Skills los leen como input; markdown es el formato nativo |
| Estado de pilares (status, timestamps, quién aprobó) | Supabase `foundation_pillars` | Datos estructurados, queryables, para MC dashboard |
| Insights extraídos durante Foundation | Supabase `insights` | Reutilizables cross-pilar, queryables |
| Historial de aprobaciones/correcciones | Supabase `foundation_approvals` | Audit trail |

**Regla**: `./brand/` = archivos markdown para consumo de skills. Supabase = datos estructurados para tracking y dashboards. Sincronizar: al aprobar un pilar, escribir el .md Y actualizar Supabase.

## Riesgos y dependencias

| Riesgo | Mitigación |
|--------|-----------|
| Flujo muy lento (14 pilares × review cada uno) | Agrupar pilares simples; L0 puede ser 2-en-1 |
| Depende de T-026 (sancho-start iterativo) | Pueden implementarse independientemente; T-026 alimenta a T-028 |
| Estado inconsistente si se editan archivos manualmente | foundation-state.json es source of truth; re-verificar al retomar |
