# PRD — T-029: GTM sigue flujo de campañas, no dump en brand/

## Objetivo

Que la ejecución de Go-To-Market siga un flujo estructurado de campañas: crear campaña → dispatch briefs → Escuderos ejecutan → Rocinante QA → publicar. `brand/` queda intocable como referencia estratégica.

## Contexto y motivación

- Actualmente cuando Sancho ejecuta GTM, genera archivos directamente en `brand/` (ej: `brand/gtm-plan.md`, `brand/content-brief-1.md`).
- `brand/` debería ser la **fuente de verdad estratégica** (Foundation): company-context, positioning, voice, competitors. No outputs de ejecución.
- Los outputs de ejecución (briefs, contenido, ads) pertenecen a **campañas**.
- Sin estructura de campañas no hay trazabilidad: ¿qué se hizo? ¿quién lo hizo? ¿cuándo? ¿qué resultados?

## Requisitos funcionales

- **RF-1**: Estructura de campañas:
  ```
  campaigns/
    2026-03-tratamientos-launch/
      brief.md          ← brief de Sancho
      tasks/             ← tareas dispatched
      content/           ← outputs de Escuderos
      qa/                ← reviews de Rocinante
      results.md         ← métricas post-ejecución
      status.json        ← estado de la campaña
  ```
- **RF-2**: Flujo de campaña:
  1. Sancho crea campaña: `campaigns/<slug>/brief.md` con objetivo, audiencia, canales, timeline
  2. Sancho despacha tareas a canales de Discord (o via sessions_spawn a Escudero)
  3. Escudero ejecuta y guarda output en `content/`
  4. Rocinante revisa (QA) y guarda feedback en `qa/`
  5. Si QA pasa → listo para publicar
  6. Post-publicación → métricas en `results.md`
- **RF-3**: `brand/` es READ-ONLY para ejecución (solo Foundation escribe ahí)
- **RF-4**: Cada campaña tiene `status.json`:
  ```json
  { "status": "in-progress", "created": "2026-03-01", "tasks": 5, "completed": 3, "qa_passed": 2 }
  ```
- **RF-5**: MC dashboard muestra campañas activas con progreso
- **RF-6**: Sancho lee `brand/` para contexto (positioning, voice) pero escribe en `campaigns/`

## Learnings integrados (2026-02-26)

### Crear skill gtm-orchestrator
No existe un orquestador de GTM. Sancho improvisó el GTM (keywords, quiz, ads, emails, briefs) porque las skills individuales existen pero no hay nada que diga "en qué orden, con qué dependencias, qué va antes de qué".

**gtm-orchestrator** debe:
- Activarse DESPUÉS del gate check de Foundation (Foundation completa y aprobada)
- Definir el flujo GTM como DAG:
  1. `keyword-research` → keywords + content pillars
  2. Validar positioning vs keywords (¿match?)
  3. `channel-prioritization` → qué canales activar
  4. Funnel architecture (T-030) → diseño del funnel
  5. `direct-response-copy` → ad copy + landing pages
  6. `email-sequences` → nurture flows
  7. `content-atomizer` → social + content distribution
  8. Creative briefs → assets necesarios
  9. Launch checklist → go/no-go
- Tener su propio state tracking (`campaigns/<slug>/gtm-state.json`)
- Referenciar archivos de Foundation como inputs READ-ONLY
- **HARD STOP** si Foundation no está aprobada: "Foundation incompleta. No puedo ejecutar GTM."

Descripción frontmatter: `"Orchestrate GTM execution: channels, funnel, copy, launch."`

## Pasos de implementación

1. Crear estructura de directorios base para campañas
2. Definir schema de brief.md y status.json
3. Actualizar BRAIN.md con reglas: brand/ = referencia, campaigns/ = ejecución
4. Modificar skills de ejecución para escribir en campaigns/ (no brand/)
5. Implementar flujo: Sancho crea brief → dispatch → Escudero ejecuta → Rocinante QA
6. Actualizar regenerate.py para leer campaigns/
7. Crear panel de campañas en MC dashboard
8. Test end-to-end con una campaña de prueba

## Criterios de aceptación

- [ ] Ningún output de ejecución se escribe en `brand/`
- [ ] Campañas tienen estructura estándar (brief, tasks, content, qa, results)
- [ ] status.json refleja progreso real de la campaña
- [ ] Flujo brief → dispatch → ejecución → QA funciona
- [ ] MC dashboard muestra campañas activas
- [ ] Sancho lee brand/ para contexto pero no escribe ahí durante GTM

## Riesgos y dependencias

| Riesgo | Mitigación |
|--------|-----------|
| Skills actuales hardcodean paths a brand/ | Buscar y reemplazar en todos los SKILL.md afectados |
| Múltiples campañas activas simultáneas | Cada campaña es un directorio independiente; no hay conflicto |
| Depende de T-015 (dispatch bot) para dispatch automático | Puede funcionar con dispatch manual (sessions_spawn) mientras T-015 no esté listo |
| Rocinante QA no está probado aún | Test aislado de Rocinante con un brief dummy antes de integrar |
