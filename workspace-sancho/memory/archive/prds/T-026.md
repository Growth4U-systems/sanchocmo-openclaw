# PRD — T-026: Sancho Start iterativo (onboarding conversacional)

## Objetivo

Convertir el skill `sancho-start` de un dump masivo de preguntas a un flujo conversacional: una pregunta → una respuesta → siguiente pregunta. El onboarding debe sentirse como una conversación con un CMO, no como rellenar un formulario.

## Contexto y motivación

- El sancho-start actual pide toda la información de golpe: empresa, producto, mercado, competidores, presupuesto, equipo...
- Esto asusta al usuario, genera respuestas superficiales ("somos una clínica capilar, no sé qué más decir"), y pierde oportunidades de profundizar.
- Un CMO real en una primera reunión no hace 20 preguntas seguidas — hace una, escucha, profundiza, y pasa a la siguiente.
- El flujo iterativo produce 10x mejor calidad de datos iniciales.

## Requisitos funcionales

- **RF-1**: Flujo paso a paso con preguntas individuales:
  1. "¿Qué hace tu empresa?" → escucha → follow-up si necesario
  2. "¿Qué producto/servicio quieres impulsar?" → escucha
  3. "¿Quién es tu cliente ideal?" → escucha
  4. "¿Quiénes son tus principales competidores?" → escucha
  5. "¿Qué presupuesto y equipo tienes?" → escucha
  6. "¿Qué has probado antes que funcionó/no funcionó?" → escucha
- **RF-2**: Cada respuesta del usuario genera follow-up inteligente si la respuesta es vaga ("¿Puedes dar un ejemplo?" / "¿Cuántos clientes al mes?")
- **RF-3**: Progreso visible: "Paso 3/6 — Vamos bien 💪"
- **RF-4**: Al completar: resumen ejecutivo de todo lo capturado, pedir confirmación
- **RF-5**: Si el usuario dice "ya te di toda la info" o similar → fast-forward con lo que hay
- **RF-6**: Guardar respuestas parciales — si la conversación se corta, poder retomar
- **RF-7**: Al terminar → escribir `brand/company-context.md` con datos estructurados y pasar a Foundation

## Archivos afectados

| Archivo | Acción |
|---------|--------|
| `skills/sancho-start/SKILL.md` | Reescribir — flujo iterativo |
| `brand/company-context.md` | Output — se genera al completar |
| `memory/onboarding-state.json` | Crear — estado parcial del onboarding |

## Pasos de implementación

1. Diseñar el flujo de preguntas: orden, follow-ups, condiciones de skip
2. Definir formato de state tracking (qué preguntas completadas, respuestas parciales)
3. Reescribir SKILL.md con instrucciones para flujo conversacional
4. Implementar detección de "respuesta vaga" → follow-up automático
5. Implementar resumen ejecutivo al final + confirmación
6. Implementar retomada de sesión (leer onboarding-state.json)
7. Test con cliente real (Hospital Capilar como referencia)

## Criterios de aceptación

- [ ] Sancho hace UNA pregunta a la vez, no un bloque
- [ ] Respuestas vagas generan follow-up inteligente
- [ ] Progreso visible en cada paso
- [ ] Resumen al final que el usuario confirma
- [ ] Sesión retomable si se interrumpe
- [ ] `brand/company-context.md` generado con datos estructurados
- [ ] El usuario siente que habló con un CMO, no rellenó un formulario

## Learnings integrados (2026-02-26)

### HARD STOP después de Phase Decision
El sancho-start actual dice "Route to /foundation-orchestrator" pero no bloquea explícitamente. Sancho lo lee, lo ignora, y ejecuta skills de Phase 2/3 directamente. Fix:

Añadir al final del Step 7 (Phase Decision):
```
⛔ STOP. No ejecutes skills de Phase 2/3 desde sancho-start.
Ruta al orchestrator correspondiente. Tu trabajo aquí terminó.
No hagas nada más. Espera a que el usuario interactúe con el orchestrator.
```

Esto es un **hard guardrail**, no una sugerencia. El SKILL.md debe usar lenguaje imperativo e inequívoco.

## Riesgos y dependencias

| Riesgo | Mitigación |
|--------|-----------|
| El flujo es demasiado lento para usuarios impacientes | Opción fast-forward: "Dame toda la info de golpe" → modo legacy |
| El modelo no mantiene estado entre turnos | Usar onboarding-state.json + inyectar contexto en cada turno |
| Follow-ups excesivos cansan | Máximo 1 follow-up por pregunta; si sigue vago, aceptar y avanzar |
