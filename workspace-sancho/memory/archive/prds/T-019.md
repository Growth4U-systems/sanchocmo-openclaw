# PRD — T-019: Health check endpoint / cron

## Objetivo

Cron job cada 6h que verifica el estado de todos los servicios del sistema. Si algo falla, alerta en #admin. Si todo OK, silencio.

## Contexto y motivación

- El sistema tiene múltiples servicios: Gateway, Discord bot, gog (Google), Notion API, Supabase, Tailscale, MC server, cron jobs.
- Actualmente Cervantes hace checks manuales en heartbeats, pero no es sistemático.
- Un cron dedicado garantiza cobertura continua sin depender del heartbeat.

## Requisitos funcionales

- **RF-1**: Cron OpenClaw cada 6h (usa `openclaw cron`)
- **RF-2**: Checks obligatorios:
  - Gateway: `openclaw status` → running
  - Discord: bot conectado y respondiendo
  - Google Workspace: `gog gmail inbox --limit 1` → no error
  - Notion: API call test → 200
  - Supabase: connection test → 200
  - Tailscale: `tailscale status` → running
  - MC server: `curl http://127.0.0.1:18790/` → 200
  - Cron jobs: verificar que los scheduled están activos
- **RF-3**: Si falla alguno → mensaje en #admin con detalle
- **RF-4**: Si todo OK → silencio (no spam)
- **RF-5**: Resultado guardado en `memory/healthcheck-state.json` con timestamps
- **RF-6**: Dashboard MC muestra último healthcheck (verde/amarillo/rojo)

## Archivos afectados

| Archivo | Acción |
|---------|--------|
| Cron OpenClaw | Crear — schedule cada 6h |
| `memory/healthcheck-state.json` | Crear/Actualizar |
| `scripts/regenerate.py` | Modificar — leer healthcheck state para MC |
| `mission-control.html` | Modificar — mostrar indicador de salud |

## Pasos de implementación

1. Definir lista de checks con comando/endpoint por servicio
2. Crear cron job OpenClaw: `openclaw cron add --every 6h --agent cervantes --task "..."`
3. Implementar lógica: ejecutar checks, recoger resultados, comparar con esperado
4. Si fallo: enviar mensaje a #admin via message tool
5. Guardar resultado en healthcheck-state.json
6. Integrar en regenerate.py → MC dashboard

## Criterios de aceptación

- [ ] Cron se ejecuta cada 6h
- [ ] Todos los servicios listados se verifican
- [ ] Fallo genera alerta en #admin con servicio + error
- [ ] OK genera silencio
- [ ] healthcheck-state.json actualizado cada ejecución
- [ ] MC dashboard muestra último estado

## Riesgos y dependencias

| Riesgo | Mitigación |
|--------|-----------|
| Check falla por timeout, no por servicio caído | Timeout generoso (30s por check), retry 1x |
| Spam en #admin si un servicio falla persistentemente | Debounce: solo alertar si cambia de estado (OK→FAIL), no en cada check |
| Cron no se ejecuta si gateway está caído | Chicken-and-egg; monitorear gateway externamente (uptime robot como P3) |
