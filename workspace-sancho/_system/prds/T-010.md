# PRD — T-010: Mission Control (Next.js) — Dual View

## Objetivo

Un solo sistema con dos puertas de entrada:
- **Vista Cliente** (`/mc/c/:slug`) — El cliente ve sus métricas, su brand, sus campañas, qué tiene pendiente de aprobar.
- **Vista Admin** (`/mc/admin`) — Alfonso, Philippe y Martín ven todo: clientes, agentes, costes, salud del sistema, y herramientas para mejorar Sancho/Cervantes.

Reemplaza el HTML estático actual + `regenerate.py`. Lectura en vivo del filesystem, escritura directa.

## Contexto

### Estado actual
- HTML monolítico (2,259 líneas) + 4 JS data files + `regenerate.py`
- Regeneración manual o cada ~2h vía heartbeat — datos siempre desfasados
- Cero interactividad: no se puede crear/editar/aprobar nada desde el dashboard
- Tailscale serve sobre archivo estático

### Por qué separar de Gateway Dashboard
- Gateway Dashboard = admin de OpenClaw (chat, nodos, config). Se actualiza con cada release de OpenClaw.
- Mission Control = operaciones de marketing + admin de SanchoCMO. Nuestro, custom, multi-tenant.
- Mezclarlos = cada `npm update openclaw` puede romper nuestros cambios.
- Integración sin fusión: MC puede leer status del gateway vía API/WS para mostrar health.

## Vista Cliente (`/mc/c/:slug`)

Lo que ve un cliente (ej: Hospital Capilar) al entrar:

### Páginas

**Dashboard** (`/mc/c/hospital-capilar`)
- Resumen ejecutivo: estado general, métricas clave, qué se ha hecho esta semana
- Progress bar de Foundation (X/14 pilares completos)
- Campañas activas con estado
- Próximos pasos: qué necesita aprobar, qué está en ejecución

**Brand** (`/mc/c/:slug/brand`)
- Documentos de marca con visualización bonita (no raw markdown)
- Pilares: company-context, positioning, voice-profile, competitors, visual-identity...
- Estado por pilar: ✅ completo / 🔧 en progreso / ⬜ pendiente
- Links directos a cada documento
- Historial de cambios en cada pilar (git log o timestamps)

**Campañas** (`/mc/c/:slug/campaigns`)
- Lista de campañas propuestas, aprobadas, en ejecución, completadas
- Click en campaña → detalle con piezas (content, ads, outreach) y estado de cada una
- Botón "Aprobar" / "Rechazar" para campañas pendientes
- Timeline visual de ejecución

**Métricas** (`/mc/c/:slug/metrics`)
- KPIs definidos en la estrategia (tráfico, leads, conversión, etc.)
- Gráficos de tendencia (cuando haya datos conectados)
- Placeholder inteligente si no hay datos: "Conecta Google Analytics para ver tráfico"

**Timeline** (`/mc/c/:slug/timeline`)
- Feed cronológico de todo lo que se ha hecho
- Filtrable por tipo: brand, campaigns, content, research
- Cada entrada enlaza al detalle

**Documentos** (`/mc/c/:slug/docs`)
- Docsify-like viewer de todos los documentos del cliente
- Buscable, navegable
- Exportable a PDF

### UX del cliente
- Landing page simple y clara. Nada técnico.
- Notificaciones: "Tienes 2 campañas pendientes de aprobar"
- Mobile responsive (el cliente mira desde el móvil)
- Acceso por email (Cloudflare Access) — cada cliente solo ve lo suyo

## Vista Admin (`/mc/admin`)

Lo que ven Alfonso, Philippe y Martín:

### Páginas

**Dashboard** (`/mc/admin`)
- Health del sistema: Gateway, Discord, Tailscale, servicios
- Costes: tokens por agente, por cliente, tendencia
- Resumen de todos los clientes (selector)
- Alertas: qué necesita atención

**Clientes** (`/mc/admin/clients`)
- Lista de clientes con estado rápido (Foundation %, campañas activas, último contacto)
- Click → entra en la vista de ese cliente (puede ver todo lo que ve el cliente + más)
- Crear nuevo cliente (ejecuta `new-client.sh`)

**Tareas** (`/mc/admin/tasks`)
- Kanban board: Propuesta → Aprobada → En Progreso → Hecha
- Drag & drop entre columnas
- Click → detalle completo + PRD si existe
- Crear tarea, aprobar/rechazar, asignar prioridad
- Filtros por categoría y prioridad
- Lee/escribe directamente en TASKS.md

**Agentes** (`/mc/admin/agents`)
- Estado de Cervantes, Sancho, Rocinante, Escudero
- Modelo actual, sesiones activas, último heartbeat
- Skills asignadas a cada uno
- Link a SOUL.md de cada agente

**Skills** (`/mc/admin/skills`)
- Las 56+ skills con búsqueda
- Click → renderiza SKILL.md con syntax highlighting
- Categorización: workspace / built-in / ClawHub
- Uso: cuántas veces se ha invocado cada skill (si trackeamos)
- Botón "Proponer cambio" → crea tarea

**Costes** (`/mc/admin/costs`)
- Dashboard de costes por agente, por cliente, por tier
- Gráficos de tendencia
- Alertas de presupuesto
- Breakdown: Opus vs Sonnet vs Haiku

**PRDs** (`/mc/admin/prds`)
- Lista de PRDs con estado
- Viewer markdown con syntax highlighting
- Crear/editar PRD

**Foundation** (`/mc/admin/foundation`)
- DAG visual de los pilares con dependencias (layers 0→5)
- Por cliente — selector
- Color por estado: ✅ done / 🔧 in progress / ⬜ pending / 🔒 blocked
- Click en pilar → contenido + botón "Lanzar desde Discord"

**Activity** (`/mc/admin/activity`)
- Feed desde memory/*.md de todos los agentes
- Filtrable por agente, por fecha
- Changelog integrado

**Configuración** (`/mc/admin/config`)
- dispatch-map.json visual (canales, roles, skills)
- Channel architecture diagram
- Editar configs (con confirmación)

### UX admin
- Sidebar con navegación completa
- Selector de cliente persistente
- Dark/light mode
- Keyboard shortcuts
- Link cruzado: "Abrir Gateway Dashboard ↗"

## Design System: Comic UI (OBLIGATORIO)

T-010 **DEBE** seguir el Comic UI Design System de SanchoCMO. La referencia completa está en `skills/comic-ui-system/SKILL.md`.

### Resumen visual

| Elemento | Especificación |
|----------|---------------|
| **Fondos** | Parchment #F5F0E6, Paper #FDF8EF, Aged #E8DCC8 (rotar, nunca repetir adyacentes) |
| **Ink/bordes** | #1A1A2E (3-4px), UI borders #2D2D44 |
| **Acentos** | Rust #C45D35, Navy #1E3A5F, Cyan #3B9EBF, Yellow #F2C94C, Sage #4A5D23 |
| **Tipografía** | Bangers (h1-h2), Comic Neue Bold (h3-h6), Comic Neue (body), Playfair italic (narraciones), Source Sans 3 (metadata) |
| **Sombras** | Flat offset sin blur: 4px 4px 0 #1A1A2E (normal), 6px 6px (hover), 2px 2px (active) |
| **Texturas** | Paper grain (SVG feTurbulence), halftone dots (radial-gradient), speed lines — todo CSS puro |
| **Componentes** | SpeechBubble, NarratorCaption, ActionBurst, ComicPanel, ComicCard, ComicButton, ComicStamp |
| **Transitions** | `ease-soft-out: cubic-bezier(0.16, 1, 0.3, 1)`, hover = shadow + translate + optional rotation |

### Reglas duras
- NO fondos grises — siempre parchment/paper/aged
- NO sombras con blur — siempre flat offset
- NO sans-serif en body text — Comic Neue para todo texto legible
- NO texturas con imágenes — todo CSS puro
- Bordes mínimo 2px (small), 3px (cards/panels), 4px (secciones)
- Navbar: parchment + 3px ink border bottom (NO glassmorphism)
- Footer: ink #1A1A2E + 3px rust top border

### Vista Cliente vs Admin — diferencia visual
- **Cliente**: Más limpio, más Comic Neue, menos densidad. Paneles grandes, CTAs claros. Se siente como un producto bonito.
- **Admin**: Más denso, más datos, más tablas. Pero mismo design system. Se siente como el backstage del mismo cómic.

### Assets reutilizables de T-013
- Logo SanchoCMO, favicon, mascota, ilustraciones empty state
- CSS variables y clases ya definidas en T-013
- T-010 hereda todo el CSS de T-013 y lo extiende con componentes React/Tailwind

### Componentes React a crear (con Comic UI)
Los componentes en `components/` deben implementar los primitivos del design system:

```tsx
// Ejemplo: ComicCard.tsx
<div className="relative bg-comic-paper border-[4px] border-comic-ink rounded-lg
  shadow-comic overflow-hidden transition-all duration-200 ease-soft-out
  hover:shadow-comic-lg hover:-translate-x-[3px] hover:-translate-y-[3px] hover:-rotate-[0.5deg]">
  {children}
</div>
```

Referencia completa de componentes: `skills/comic-ui-system/references/component-api.md`
