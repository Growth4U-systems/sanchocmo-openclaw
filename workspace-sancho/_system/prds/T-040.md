# T-040 — Intelligence Log

## Problema
Los crons de Meeting Intelligence y Daily Pulse no tenían histórico centralizado. Las reuniones se reprocesaban. No había visibilidad en MC de qué intelligence se había procesado.

## Solución
**Intelligence Log** — archivo JSON centralizado (`_system/intelligence-log.json`) que:
1. Registra todo item de inteligencia procesado (meetings, pulses, research futuro)
2. Sirve como fuente de deduplicación para crons
3. Alimenta una sección visual en Mission Control

## Schema
```json
{
  "version": 2,
  "entries": [
    {
      "id": "mtg-{slug}|pulse-{date}",
      "type": "meeting|daily-pulse|research|thief-marketer",
      "client": "{client-slug}",
      "date": "YYYY-MM-DD",
      "title": "Título descriptivo",
      "summary": "Resumen corto (max 200 chars)",
      "status": "processed",
      "sourceFile": "ruta relativa al archivo fuente",
      "processedAt": "ISO timestamp",
      "tags": ["meeting", "daily", ...]
    }
  ]
}
```

## Componentes
1. `_system/intelligence-log.json` — fuente de verdad
2. `regenerate.py` → `parse_intelligence_log()` — parsea y expone en mc-data.js
3. MC `renderIntelligence()` — tabla con filtros por tipo, búsqueda, links a docs
4. Skills actualizados — meeting-intelligence y daily-pulse escriben al log
5. Dedup — crons consultan el log antes de procesar
