# PRD — T-014: Panel de APIs en Mission Control (globales + por cliente)

## Objetivo

Mostrar en Mission Control el estado de todas las APIs y servicios que usa SanchoCMO, separados en dos niveles:
- **Globales**: APIs compartidas del sistema (Anthropic, OpenRouter, Brave, Apify, OpenClaw, Google Workspace)
- **Por cliente**: APIs específicas de cada cliente (Google Ads, Meta Ads, GA4, Search Console, Apollo, etc.)

El panel permite saber de un vistazo qué está conectado, qué falta, y qué tiene errores — sin revisar configs manualmente.

## Contexto y motivación

MC tiene una sección "📡 Integraciones" que muestra integraciones por cliente desde `integrations.json`, pero:
1. **No muestra APIs globales** — No se ve si Anthropic, Brave o Apify están funcionando
2. **No hay health check** — Solo dice "connected/none" sin verificar si el token sigue válido
3. **No hay distinción clara** entre sistema y cliente
4. **Falta visibilidad de consumo** — No se ve el uso de cada API

## Requisitos funcionales

### Vista Global (admin)

| API | Datos a mostrar |
|-----|----------------|
| **Anthropic** | Estado (ok/error), modelo default, tokens últimas 24h, coste estimado |
| **OpenRouter** | Estado, modelos disponibles, tokens últimas 24h, coste |
| **Brave Search** | Estado, queries últimas 24h |
| **Apify** | Estado, token válido, créditos restantes |
| **Google Workspace** | Estado (gog auth), cuenta (alfonso@growth4u.io) |
| **OpenClaw Gateway** | Versión, uptime, agentes activos, sesiones |

Cada fila muestra:
- 🟢 Conectado / 🟡 Sin verificar / 🔴 Error / ⚪ No configurado
- Última verificación (timestamp)
- Botón "🔄 Verificar" para health check bajo demanda

### Vista Cliente

Reutiliza la sección actual de integraciones (`integrations.json`) pero añade:
- Indicador de health check (no solo status estático)
- Columna "Último uso" (cuándo Sancho usó esa API para ese cliente)
- Enlace a setup si `status === "none"`

### Health Check

Endpoint nuevo en mc-server: `GET /api/health-check?service=<id>`

Verificaciones por servicio:
- **Anthropic**: curl a API con key → 200 = ok
- **Brave**: query de prueba → respuesta = ok
- **Apify**: GET /v2/user/me con token → 200 = ok
- **Google Workspace**: `gog me` → respuesta = ok
- **OpenClaw**: `openclaw status` → parse output
- **Servicios de cliente**: marcar como "pending" (requiere credenciales)

### Datos persistentes

Archivo: `_system/api-health.json`

```json
  "lastCheck": "2026-02-27T13:00:00Z",
  "services": {
    "anthropic": { "status": "ok", "lastCheck": "...", "details": { "model": "claude-opus-4-6" } },
    "brave": { "status": "ok", "lastCheck": "...", "details": {} },
    "apify": { "status": "ok", "lastCheck": "...", "details": { "creditsRemaining": 42.5 } },

    "openclaw": { "status": "ok", "lastCheck": "...", "details": { "version": "2026.2.26", "agents": 4 } }
  }
}
```

### Cron de health check

Nuevo cron o añadir a heartbeat: verificar APIs globales 2x/día. Guardar en `api-health.json`. Si fallo → notificar en #admin.

## Pasos de implementación

1. **Crear `_system/api-health.json`** con estructura inicial (todos "unchecked")
2. **Implementar health checks** en mc-server.js:
   - `GET /api/health-check?service=all` — ejecuta todos
   - `GET /api/health-check?service=anthropic` — ejecuta uno
   - Cada check: exec comando, parsear output, guardar en json
3. **Añadir sección "🔌 APIs" en MC** (vista global):
   - Grid de cards (una por API global)
   - Badge de status (🟢/🟡/🔴/⚪)
   - Botón "Verificar todo"
4. **Mejorar sección integraciones** (vista cliente):
   - Badge de health check junto al status actual
   - Mostrar `lastUsed` si existe
5. **Actualizar `regenerate.py`** para incluir api-health en mc-data.js
6. **Añadir health check a HEARTBEAT.md** (2x/día)

## Criterios de aceptación

- [ ] Vista global muestra todas las APIs del sistema con status real
- [ ] Health check funciona (Anthropic, Brave, Apify, gog, OpenClaw)
- [ ] Botón "Verificar" ejecuta check y actualiza UI sin reload
- [ ] Vista cliente mantiene integraciones actuales + añade health badges
- [ ] `api-health.json` se actualiza con cada check
- [ ] Si una API falla, se notifica en Discord #admin
- [ ] Comic UI aplicado (parchment, Bangers headers, ink borders)

## Riesgos y dependencias

| Riesgo | Mitigación |
|--------|-----------|
| Health checks exponen API keys en logs | Checks server-side (mc-server), nunca frontend |
| Rate limits de Anthropic/Brave | Max 2 checks/día automáticos + manual bajo demanda |
| `gog` no está en PATH de mc-server | Usar ruta absoluta |
| Latencia de checks bloquea UI | Async, UI muestra spinner |
| Coste por health check | Queries mínimas |
