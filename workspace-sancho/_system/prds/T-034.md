# T-034 — Integraciones y Costes por Cliente en Mission Control

## Resumen

Añadir a Mission Control (estático) una sección de integraciones y costes por cliente. Cada cliente tiene un `integrations.json` que lista sus APIs conectadas y su estado. MC lo lee y muestra una tabla visual. El cost tracker escribe costes desglosados por cliente.

- Tabla con: Servicio | Estado (badge de color) | Cuenta | Última actividad

- Debajo: resumen de costes del mes

### 4. Mission Control — Vista global (admin)

En la vista global, tabla resumen:

- Cliente | Coste SanchoCMO | APIs activas | Estado general
- Total en la última fila

### 5. regenerate.py

Actualizar `regenerate.py` para que genere un `integrations-data.js` que MC pueda leer, igual que hace con `mc-data.js`.

## Credenciales

Las credenciales reales (tokens, API keys) NO van en `integrations.json`. Van en:
- `openclaw.json` → sección de skills con apiKey
- Variables de entorno del gateway

`integrations.json` solo tiene metadata (estado, accountId, notas). Nunca secrets.

## Ficheros a crear/modificar

| Fichero | Acción |
|---------|--------|
| `brand/hospital-capilar/integrations.json` | Crear — seed con servicios iniciales |
| `brand/hospital-capilar/costs.json` | Crear — seed vacío |
| `mission-control.html` | Modificar — añadir sección integraciones + costes |
| `scripts/regenerate.py` | Modificar — generar integrations-data.js |
| `scripts/mc-server.js` | Modificar — servir integrations-data.js |

## Estimación

- 1-2h de implementación
- Solo CSS/HTML/JS en MC — sin backend nuevo
- Datos estáticos desde ficheros JSON

## Decisiones

- Credenciales las pedimos al cliente (no OAuth por ahora)
- `integrations.json` por cliente, no global
- Cost tracker (T-022) se integrará después para escribir `costs.json` automáticamente
- Fase 2 (T-010 Next.js) migrará a Supabase
